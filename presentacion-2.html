El problema que resolvemos

Aprender Simplex es difícil:
las respuestas solas no enseñan

  • Las calculadoras de LP dan el resultado, pero ocultan el proceso.
  • Un LLM solo, "a ojo", inventa números y se equivoca en la aritmética.
  • El estudiante necesita ver cada iteración: variable entrante, saliente, pivote.

Nuestra apuesta: separar el cálculo exacto (un solver determinista) de la explicación pedagógica (el LLM). Lo mejor de los dos mundos.

Cómo funciona

El flujo de una consulta

1
El usuario plantea el problema → useChat hace POST /api/chat.
2
El servidor valida la sesión y hace upsert del chat.
3
gpt-5.5 confirma que entendió y pregunta: "¿Lo resuelvo con Simplex?" — todavía no calcula.
4
El usuario confirma → el modelo decide invocar la tool solveSimplex con el problema en JSON.
5
simplex.ts ejecuta Big-M y devuelve los tableaux de cada iteración.
6
El modelo explica iteración por iteración; los tokens llegan por streaming y KaTeX renderiza el LaTeX al vuelo.
7
onFinish persiste todo en Supabase y genera el título con gpt-4.1-mini.
Decisión de diseño · la más importante

El LLM nunca calcula los números

  • El modelo siempre invoca al solver y usa sus resultados; no hace aritmética por su cuenta.
  • Esto elimina las alucinaciones numéricas, el punto débil de los LLM en matemática.
  • El solver entrega un snapshot del tableau por iteración: insumo perfecto para que el modelo "lea la pizarra" y explique.
  • Resultado: exactitud determinista + explicación natural.
// Contrato entre LLM y solver (Zod) { opType: "max" | "min", objective: { x1: 3, x2: 5 }, constraints: [ { coefficients: { x1: 1, x2: 2 }, type: "<=", rhs: 4 } ] } // → { feasible, bounded, optimal, // variables, tableaux[] }
El núcleo matemático

Solver Simplex Big-M, escrito a mano

  • Forma estándar automática: ≤ agrega holgura · ≥ agrega exceso + artificial · = agrega artificial.
  • Penalización Big-M = 1.000.000 fuerza la salida de las variables artificiales.
  • Iteración: entrante por costo reducido más negativo · saliente por razón mínima · pivoteo por operaciones de fila.
  • Detecta infactibilidad y no acotamiento. Sin dependencias externas para la matemática.

Persistencia de datos

Base de datos y guardado

  • El guardado ocurre en onFinish tras cerrar el stream — consumeStream() asegura que corra aunque el cliente se desconecte.
  • Estrategia delete + re-insert de todos los mensajes: estado consistente ante reintentos.
  • Mensajes guardados como JSONB (texto, imágenes, tool calls y resultados).
  • Índices por (user_id, updated_at) y (chat_id, order) para cargas rápidas.

Seguridad

Tres capas de protección

RLS

Row Level Security

RLS en toda la base: políticas SELECT/INSERT/UPDATE/DELETE garantizan que cada usuario solo ve y modifica sus propios datos (auth.uid() = user_id).

Proxy

Middleware de rutas

proxy.ts verifica la sesión en cada request a /chat/*. Sin sesión válida → redirect a /login antes de procesar la ruta.

API

API protegida

/api/chat valida con getUser() antes de procesar; sin usuario responde 401. Las claves secretas viven solo en el servidor.

Resumen

Decisiones de diseño que tomamos

Solver ≠ LLM

Separar cálculo exacto de explicación: cero alucinaciones numéricas.

Big-M, no dos fases

Maneja todos los tipos de restricción en un solo paso, más simple de explicar.

Confirmación previa

El modelo confirma el modelo matemático antes de resolver: refuerza el aprendizaje.

stepCountIs(5)

Límite del ciclo LLM→tool→LLM para evitar bucles en problemas complejos.

PDF como fuente

El Cap. 3 de Taha se inyecta como contexto para anclar la teoría y citarla.

Server + Client Components

El historial se hidrata en el servidor; el chat es interactivo en el cliente.

Honestidad técnica

Limitaciones conocidas

  • Asume variables de decisión no negativas (x ≥ 0).
  • No soporta variables libres (sin restricción de signo).
  • El RHS de cada restricción debe ser ≥ 0.
  • Sin regla de Bland: problemas degenerados podrían ciclar (corte a 200 iteraciones).
  • Alcance acotado a propósito: solo método Simplex.

Son límites conscientes y documentados: el objetivo es enseñar Simplex en problemas educativos, no ser un solver industrial de propósito general.
